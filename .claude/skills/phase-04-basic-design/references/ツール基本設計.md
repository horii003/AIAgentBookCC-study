# 役割
あなたはAIエージェント開発の基本設計を専門とする設計者です。

# 指示
システム要件定義とシステム設計の成果物をもとに、フォーマットに従ったツール基本設計書を作成してください。

# 設計対象ツールの絞り込み（必須・最初に実施）

基本設計書を作成する前に、`機能ツール一覧.md` の各TOOLについて以下の手順で設計対象を絞り込むこと。

**手順**:
1. `システム基本情報.md` の技術スタックセクションを参照し、採用フレームワーク・ライブラリの一覧を確認する
2. 各TOOLについて、以下の除外判定を順番に確認する

**除外判定（いずれかに該当するTOOLは基本設計書の対象外とする）**:

- **フレームワーク・ライブラリの組み込み機能で実現できる**: 採用フレームワークまたはライブラリが標準で提供する機能（例：`strands_tools` の `image_reader` 等）で要件を満たせる場合は対象外とする
- **外部ライブラリの薄いラッパーになる**: ライブラリが提供する関数を呼び出すだけで、独自のビジネスロジックを持たない場合は対象外とする
- **ナレッジ参照がシステムプロンプト埋め込みで実現できる**: 採用フレームワークの仕組み上、ナレッジ内容をシステムプロンプトに直接埋め込む方式で実現できる場合は対象外とする（RAGや外部ナレッジベースAPIが必要な場合のみ対象とする）

3. 除外したTOOLは設計書を作成しない
4. 除外されなかったTOOLのみを対象として基本設計書を作成する

# 入力データの優先順位
システム設計フェーズ（SD系）とシステム要件定義フェーズ（SR系）で内容が相反する場合は、システム設計フェーズの成果物を正とすること。

# 前提条件
- フォーマットの項目・構成・順序を厳守すること
- システム要件定義・システム設計から読み取れる内容のみを端的にポイントをおさえて記載すること
- 判断できない内容は「要件上未定義」と記載すること
- 解説や補足説明は記載しないこと
- 参照元セクションには、実際に読み込んだ成果物ファイルのパス（artifacts/配下）を記載すること。テンプレートファイル名をそのまま転記してはならない
- ツールIDはTL-XXX形式で付与すること
- ツール種別（情報取得／実行／データ処理）を明確に分類すること
- エージェント視点での入出力設計を重視すること
- 複数エージェントで使用する場合のみ「10. マルチエージェント連携時の扱い」を記載すること
- 補助関数がある場合のみ「13. 補助機能」を記載すること
- ツール名（関数名）は英語で記載すること
- ファイル名およびタイトルの[ツール名]は、機能ツール一覧.mdの「ツール名」列の値をそのまま使用すること（省略・変形・英語変換・英語識別子への置き換え禁止）

## 各項目の記載ガイド

### 基本情報
- ツールIDはTL-XXX形式で記載すること（システム要件定義の識別子と一致させること）
- ツール種別は「情報取得」「実行（成果物生成）」「データ処理」から選択して記載
- ツール名（関数名）は英語で記載

### 目的・役割定義
- エージェント単独では実現できない理由を明確に記載すること
- 主責務は箇条書きで端的に記載
- 非責務は担当コンポーネント名を明記して記載
- 設計意図（なぜその設計にしたか）を記載すること

### 利用エージェント
- 利用エージェントのエージェントIDと利用目的を記載すること
- 利用タイミング（会話フローのどのステップか）を記載

### ツール利用トリガー設計
- 業務ステップと利用判断条件を一覧化すること
- エージェントがツールを呼び出す条件を明確に記載

### 入力設計
- エージェント視点での入力の意味・取得方法・不足時対応を記載すること
- 必須パラメータと任意パラメータを区別して記載
- バリデーション方針.mdに準拠した入力検証ルールを記載

### 出力設計
- エージェント視点での出力の意味・利用方法・信頼性を記載すること
- 正常時の出力形式と内容を記載
- エージェントが出力をどのように解釈・利用するかを記載

### ツール結果の解釈ルール
- 正常時・エラー時の判断と次アクションを記載すること
- エージェントが結果をどのように判断するかを記載

### フォールバック・代替設計
- 再試行条件と人間への引き渡し条件を記載すること
- 実行制御方針.mdに準拠したリトライ方針を記載

### エラー・例外処理設計
- 例外処理方針.mdに準拠したエラー対応を記載すること
- エラー種別ごとの対応方針を記載

### マルチエージェント連携時の扱い（複数エージェントで使用する場合のみ）
- 複数エージェントで使用する場合のみ記載すること
- ツール共有時の競合防止方針を記載

### 詳細設計への引き渡し事項
- ツール名（関数名）・デコレータ・実装注意点を記載すること
- 詳細設計で決定すべき事項を明記
- **ツール分割方針を必ず決定すること**：機能ツール一覧で1つのツールとして定義されていても、以下のチェックステップに従って分割要否を判断し、最終的なツール関数名と数をここで確定すること

  **【チェックステップ】以下を順番に確認すること**

  1. 呼び出しエージェントが1つ → **単一ツール**（以降のステップ不要）
  2. 各エージェントが渡す必須パラメータのフィールドをデータモデル基本設計書で確認し比較する
     - 引数名が同じでも `dict`/`list`/`Union` 型は内側のモデル・フィールドまで確認すること
     - フィールドセットが異なる → **分割する**
     - フィールドセットが同一 → 次へ（「次へ」行く前に、下記の「禁止：以下の理由による「単一ツール」判断は認めない」を再度確認する）
  3. 各エージェントが受け取る出力の構造・項目が同一か確認する
     - 同一 → **単一ツール**（内部分岐可）
     - 異なる → **分割する**

  > 種別パラメータによる内部分岐は、ステップ2・3を通過した（全エージェントで入出力が共通）場合のみ許容する

  > **⛔ 禁止：以下の理由による「単一ツール」判断は認めない**
  >
  > - 「`dict` 型や汎用コンテナ型（`Any`、`object`など）で包めばフィールドが異なっていても共通インターフェースにできる」という理由での単一ツール化
  > - 「種別パラメータ（`template_type`、`mode`、`type`など）を追加すれば内部分岐で対応できる」という理由での単一ツール化（ステップ2・3を通過していない場合）
  > - 「呼び出し元が渡す値の意味・構造が異なっていても、引数名を同じにすれば共通化できる」という表面的な一致による単一ツール化
  >
  > **判断の原則**: 「共通インターフェースに統一できるか」ではなく「各エージェントが渡す実際のフィールドセットが同一か」で判断すること。フィールドセットが異なる場合は、どれだけ実装上の工夫で吸収できても**必ず分割する**。

### 補助機能（補助関数がある場合のみ）
- 補助関数がある場合のみ記載すること
- 補助関数名（英語）と目的の概要を記載

### 他設計書との対応関係
- 関連する全設計書を表形式で列挙すること
- 本設計書がどの設計書の何を参照・提供するかを明記すること
- 上流（参照元）と下流（参照先）の両方を記載すること
