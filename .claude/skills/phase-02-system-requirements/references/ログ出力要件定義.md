# 役割
あなたはAIエージェント開発のシステム要件定義を専門とする設計者です。

# 指示
業務要件定義とエージェント一覧をもとに、フォーマットに従ったログ出力要件定義書を作成してください。

# 前提条件
フォーマットの項目・構成・順序を厳守すること
業務要件から読み取れる内容のみを端的にポイントをおさえて記載すること
判断できない内容は「要件上未定義」と記載すること
解説や補足説明は記載しないこと
監査ログとデバッグログは混在させないこと
内部推論（Chain-of-Thought）は保存しないこと
すべてのログにrequest_idを含めること（相関ID設計）
機密区分に応じたマスキング要件を定義すること

## 2. 各項目の記載ガイド

### ログ分類
- 監査ログ（AUD-xxx）：説明責任・証跡
- 操作ログ（OPE-xxx）：利用状況把握
- トレースログ（TRC-xxx）：不具合解析
- エラーログ（ERR-xxx）：障害検知

### ログIDの採番ルール
- 監査ログ：`AUD-001`
- 操作ログ：`OPE-001`
- トレースログ：`TRC-001`
- エラーログ：`ERR-001`

### 相関ID設計（必須）
- request_id：会話・処理全体の相関
- step_id：会話ステップ
- tool_call_id：ツール呼出単位
- すべてのログに request_id を含めること

### マスキング要件
- 個人情報・機密情報は必ずマスキング対象として定義する

# 受入基準
作成後に以下の受入基準を満たしているか確認すること。
- 監査ログ（AUD-xxx）・操作ログ（OPE-xxx）・トレースログ（TRC-xxx）・エラーログ（ERR-xxx）が定義されていること
- 監査ログとデバッグログが混在していないこと
- Chain-of-Thought（内部推論）が保存対象に含まれていないこと
- すべてのログにrequest_idが含まれること、step_id・tool_call_idが定義されていること
- 個人情報・機密情報のマスキング要件が定義されていること
- 各ログにAUD-xxx/OPE-xxx/TRC-xxx/ERR-xxx形式のIDが付与されていること
- テンプレートの全セクション・項目が含まれていること
- 業務要件から読み取れる内容のみが記載されていること（推測・補完なし）
