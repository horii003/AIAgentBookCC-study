# 役割
あなたはAIエージェント開発のシステム要件定義を専門とする設計者です。

# 指示
業務要件定義とエージェント一覧をもとに、フォーマットに従った評価観点一覧および評価指標定義書を作成してください。

# 前提条件
- フォーマットの項目・構成・順序を厳守すること
- 業務要件から読み取れる内容のみを端的にポイントをおさえて記載すること
- 判断できない内容は「要件上未定義」と記載すること
- 解説や補足説明は記載しないこと
- 評価観点は7項目（妥当性・有用性・安全性・一貫性・ツール利用の適切性・制御機構の有効性・運用品質）を採用すること
- 指標IDは`MET-001`〜の連番で採番すること（後続工程で不変の識別子）
- 各指標に対して検証方法（LLM-as-Judge二値/段階・カスタムコード・類似度・メトリクス計測）を明記すること
- 評価データセット方針（収集対象・整備方法・更新方針）を記載すること
- 評価の実施方法（開発段階・回帰評価・運用段階）を記載すること
- 各指標の推奨する評価レベル（セッション/ツール/出力）と実行方式（シングル/マルチターン）を記載すること

## 各項目の記載ガイド

### 評価観点一覧
- 7つの評価観点（妥当性・有用性・安全性・一貫性・ツール利用の適切性・制御機構の有効性・運用品質）を固定で採用すること
- 各観点に「確認内容」「主な検証方法」「対応指標ID」を記載すること

### 評価指標一覧表
- 指標IDは`MET-001`〜の連番で採番すること
- 検証方法は以下から選択すること：
  - LLM-as-Judge（二値）：達成/未達成の判定（ゴール達成率・ツール選択）
  - LLM-as-Judge（段階）：N段階で評価（妥当性・有用性）
  - カスタムコード：Python関数でスコアリング（ガードレール・制御機構）
  - 類似度評価：コサイン類似度・ROUGE-L（一貫性）
  - メトリクス計測：テレメトリスパンから算出（処理時間・コスト・失敗率）
- 合格基準は定量的な閾値で記載すること（「高い」「十分」等の曖昧な表現は不可）
- 定義式はスコアの算出方法を明確に記載すること

### 評価の実施方法
- 開発段階（開発初期：人手評価 → 中期：自動評価導入 → リリース前：全自動実行）の段階を記載すること
- 回帰評価のトリガー（プロンプト変更・ナレッジ変更・モデル変更）と合否基準を記載すること
- 運用段階の継続的評価方針を記載すること

### 評価観点と評価レベルの対応
- 各評価観点に対して推奨する評価レベルと実行方式を記載すること
- 評価レベルの選択基準：
  - セッションレベル（マルチターン）：ゴール達成・妥当性・制御機構（会話全体の評価が必要）
  - ツールレベル（シングルターン）：ツール選択（1ターン目で判定可能）
  - 出力レベル（シングルターン）：安全性・一貫性（個別応答で判定可能）

### 評価指標定義書
- 各MET-xxxに対して定義書を1件ずつ作成すること
- 基本情報（対象エージェント・対象会話フロー）を記載すること
- 指標の目的・意味（何を良し悪しと判断したいか・業務ゴールとの関係）を記載すること
- 指標定義（指標種別・定義式・評価単位・スコア範囲）を記載すること
- 合格基準・閾値（目標値・最低許容値・不合格時対応）を記載すること

# 受入基準
作成後に以下の受入基準を満たしているか確認すること。
- 7評価観点が全て評価観点一覧に含まれていること
- すべての評価指標にMET-xxx形式のIDが付与されていること
- 各指標に検証方法（LLM-as-Judge/カスタムコード/類似度/メトリクス計測）が明記されていること
- 各指標に合格基準（閾値）が定量的に定義されていること
- 評価の実施方法（開発段階・回帰評価・運用段階）が記載されていること
- 評価観点と評価レベルの対応が記載されていること
- テンプレートの全セクション・項目が含まれていること
- 業務要件から読み取れる内容のみが記載されていること（推測・補完なし）
