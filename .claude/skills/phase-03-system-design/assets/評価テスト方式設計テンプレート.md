## 1. 目的・適用範囲

### 1.1 目的

AIエージェントシステムの品質を保証するための評価テスト方式を定義する。本書は詳細設計フェーズの評価テスト詳細設計および実装フェーズの評価テストコード生成へのインプットとなる。

> **評価指標・合否基準・実施方法・回帰評価方針は `評価指標一覧.md` を正とする。本書はそれらを参照し、評価の「仕組み・アーキテクチャ・環境」を定義する。**

### 1.2 適用範囲

- AIエージェント評価テスト：LLM-as-Judge方式によるエージェント品質の自動評価

### 1.3 本方針で定義しないこと

- 個別テストケースの入力データと期待結果（詳細設計で定義）
- 評価テストコードの実装詳細（実装フェーズで定義）
- 評価フレームワーク固有のAPI仕様（詳細設計で定義）
- 評価指標の定義・閾値・実施方法・回帰評価方針（評価指標一覧.mdで定義）

## 3. 評価テストの共通抽象概念

### 3.1 コア抽象概念の定義

| 抽象概念 | 定義 |
|---------|------|
| テストケース | 評価の単位。入力・期待出力・期待軌跡・メタデータを持つ |
| 評価方式 | テストケースの実行結果に対してスコアリング（LLM-as-Judge・軌跡マッチ等）を適用し、スコア・合否・判定理由を返す評価の方法 |
| 評価結果 | スコア (float 0-1) + 合否 (bool) + 判定理由 (str) |
| 実験 | テストケース群 × 評価方式群のバッチ実行単位 |
| レポート | 全体スコア・個別スコア・判定理由の集約 |

### 3.2 テストケースの構成要素

| 要素 | 型 | 必須/任意 | 説明 |
|------|---|---------|------|
| 識別名 | string | 必須 | テストケースの一意識別子 |
| 入力 | string / messages | 必須 | ユーザー発話またはメッセージ配列 |
| 期待出力 | string | 任意 | 期待する最終応答 |
| 期待軌跡 | list[string] | 任意 | 期待するツール呼び出し順序 |
| メタデータ | dict | 任意 | 参照する追加情報 |
| セッションID | string | 自動生成 | テレメトリ分離用 |

### 3.3 評価方式の種類

| 評価方式種別 | スコアリング方式 | 適用場面 |
|-----------|---------------|---------|
| **LLM-as-Judge（二値）** | 達成/未達成 → 1.0/0.0 | ゴール達成率・ツール選択妥当性 |
| **LLM-as-Judge（段階）** | N段階評価 → 0.0〜1.0 | 応答品質・有用性・一貫性 |
| **軌跡マッチ（決定的）** | ツール呼び出し列の一致判定 | ツール選択の正確性 |
| **完全一致/文字列マッチ** | 出力文字列の一致判定 | 分類タスク・固定回答 |
| **類似度評価** | コサイン類似度・ROUGE-L | 回答の一貫性・要約品質 |
| **カスタムコード** | Python関数によるスコアリング | 業務固有ルール・メトリクス計測 |

### 3.4 軌跡マッチモード

| マッチモード | 判定基準 | 適用場面 |
|-----------|---------|---------|
| strict | 完全一致（順序含む） | 厳密なワークフロー検証 |
| unordered | 順不同で全要素一致 | 順序不問のツール呼び出し検証 |
| subset | 期待が実際の部分集合 | 最低限のツール呼び出し検証 |
| superset | 実際が期待の部分集合 | 不要なツール呼び出しの検出 |

## 4. 評価レベルの定義

### 4.1 評価レベル体系

| 評価レベル | 評価対象 | 評価方式 | 判定粒度 | 対応指標 |
|-----------|---------|---------|---------|---------|
| セッションレベル | 会話全体のゴール達成 | LLM-as-Judgeがセッション全体の軌跡を評価 | セッション単位 | [MET-xxx] |
| ツールレベル | 個別のツール選択判断 | LLM-as-Judgeが各ツール呼び出しの妥当性を評価 | ツール呼び出し単位 | [MET-xxx] |
| 出力レベル | 最終応答の品質 | LLM-as-Judge / 類似度 / 完全一致 | 応答単位 | [MET-xxx] |
| 軌跡レベル | ツール呼び出し列全体 | 決定的マッチ / LLM-as-Judge | 軌跡単位 | [MET-xxx] |

### 4.2 評価レベルの選択基準

| 確認したいこと | 選択すべきレベル | 実行方式 |
|-------------|---------------|---------|
| ユーザーの業務ゴールが達成されるか | セッションレベル | マルチターン |
| 正しいツールが選択されるか | ツールレベル | シングルターン |
| 最終応答の品質 | 出力レベル | シングルターン |
| ツール呼び出しの順序・網羅性 | 軌跡レベル | シングル or マルチターン |

## 5. 評価方式の共通アーキテクチャ

### 5.1 実行フロー

```
┌─────────────────────────────────────────────────────────┐
│                    評価実行基盤                            │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  [1. テストケース定義]                                    │
│    - 入力（ユーザー発話）                                  │
│    - メタデータ（ゴール / 期待ツール / ルーブリック）         │
│                                                         │
│  [2. タスク実行]                                         │
│    - エージェントにテストケースの入力を送信                  │
│    - テレメトリによる実行軌跡（スパン）の自動記録            │
│    - シングルターン or マルチターン（ユーザーシミュレーター）  │
│                                                         │
│  [3. 評価判定]                                           │
│    - 評価方式が実行結果（出力 + 軌跡）をスコアリング          │
│    - LLM-as-Judge / 決定的マッチ / カスタムコード           │
│                                                         │
│  [4. 結果レポート]                                        │
│    - 全体スコア（overall_score）                          │
│    - 個別スコア・合否・判定理由                            │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 5.2 テストケース設計方針

| 項目 | 方針 |
|------|------|
| テストケースの構成要素 | 識別名・入力テキスト・タスク説明・判定基準 |
| テストデータの独立性 | 各テストケースは他のケースに依存しない |
| テレメトリの分離 | ケース実行前にスパンをクリアし、ケース間のデータ混入を防止 |
| 評価データセットの拡充 | 運用開始後も実際の利用データをもとに継続的にケースを追加 |

### 5.3 ユーザーシミュレーター方式

| 項目 | 方針 |
|------|------|
| シミュレーターの役割 | ゴールに基づき動的なユーザー発話を生成 |
| 最大ターン数 | [デフォルト値]ターン |
| 会話終了条件 | ゴール達成判定 or 最大ターン到達 |
| 入力の起点 | テストケースの入力フィールド |

### 5.4 LLM-as-Judge設計方針

| 項目 | 方針 |
|------|------|
| 判定モデル | 評価対象とは別モデルを推奨 |
| 判定根拠の出力 | 判定理由を必ず出力しレポートに含める |
| スコアリング尺度 | 二値を基本。段階評価はルーブリック明示 |
| Few-shot例 | 複雑な判定にはfew-shot exampleを活用 |

## 8. 評価実施フロー

### 8.1 実施工程

```
詳細設計完了
  ↓
評価テスト実施（python evals/eval_*.py）
  ↓
評価指標測定・合否判定
  ↓
品質チェック通過
```

### 8.2 ゲート条件（合格基準）

> 各MET-xxxの閾値は `評価指標一覧.md` の各指標定義書「5. 合格基準・閾値」を参照。

| ゲート | 移行基準 |
|--------|---------|
| 評価テスト完了 | 評価指標一覧.mdに定義された全指標の最低許容値を達成すること |
| 品質チェック完了 | 評価指標全項目が合否基準を満たすこと |
