# 役割
あなたはAIエージェント開発の例外処理設計を専門とする設計者です。

# 指示
システム要件定義の成果物をもとに、フォーマットに従った例外処理方針を作成してください。

# 前提条件
- フォーマットの項目・構成・順序を厳守すること
- システム要件定義から読み取れる内容のみを端的にポイントをおさえて記載すること
- 判断できない内容は「要件上未定義」と記載すること
- 解説や補足説明は記載しないこと
- 例外分類（EX-01〜EX-08）を本システムの具体例で埋めること
- 例外×対応方針マトリクスを本システムの業務ルールに基づいて定義すること
- エスカレーション対象と判定責務を明確にすること
- UX観点（ユーザーへの通知・エラーメッセージ抽象度）を必ず記載すること

## 2. 各項目の記載ガイド

### 例外処理の基本原則
- システム共通の5原則（無限リトライ禁止・状態管理・即エラー表示しない・内部失敗を見せない・後続処理に影響を残さない）を本システムに適用

### 例外分類
- EX-01〜EX-08の各分類に本システム固有の具体例を記載
- 業務ルール違反（業務要件で定義された制約条件）を必ずEX-03に含めること

### 例外検知レベル
- 業務レベル・処理レベル・システムレベルの検知主体を明確化

### 例外×対応方針マトリクス
- 再試行/代替処理/エスカレーション/中断の組み合わせを本システムの業務ルールに基づいて定義
- 各セルの記号凡例：`○`（実施する）、`×`（実施しない）、`△`（条件付きで実施）
- 代替処理・備考欄には具体的な対応内容（例：再入力を促す、質問返しをする）を記載すること

### ErrorHandlerの責務方針
- メソッド名・引数・戻り値などの実装レベルの情報は記載しない
- ErrorHandlerが担う責務（継続可否判定・ユーザー向けメッセージ生成・ログ出力）を方針レベルで記述する
- 具体的なメソッド設計は基本設計（ハンドラー基本設計.md）で定義する旨を明記する

### エラーメッセージ設計方針
- ユーザーフレンドリーな日本語メッセージの方針を定義
- 具体的なメッセージ例を記載

### エスカレーションとの関係
- エスカレーション対象例外と行き先を定義
- 判定責務（ErrorHandler）を明記

### セッション状態への反映方針
- 例外時の状態更新（ACTIVE/WAITING/TERMINATED）と再開可否を定義
