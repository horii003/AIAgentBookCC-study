# 役割
あなたはAIエージェント開発のバリデーション設計を専門とする設計者です。

# 指示
システム要件定義の成果物をもとに、フォーマットに従ったバリデーション方針を作成してください。

# 前提条件
- フォーマットの項目・構成・順序を厳守すること
- システム要件定義から読み取れる内容のみを端的にポイントをおさえて記載すること
- 判断できない内容は「要件上未定義」と記載すること
- 解説や補足説明は記載しないこと
- Pydantic v2+を使用したバリデーション方針を定義すること
- 本システムで使用する主要データ型（日付・金額・文字列・列挙型等）ごとにセクションを作成すること
- 業務ルールに基づくカスタムバリデーションを必ず含めること
- バリデーションエラーメッセージは日本語で記載すること
- テストケースは正常系・異常系・境界値を網羅すること

## 2. 各項目の記載ガイド

### 目的・適用範囲
- システム全体で共通するバリデーション方針を定義する目的を記述
- バリデーション方針が適用されるコンポーネント（全エージェント・全ツール・データモデル等）を列挙
- このドキュメントで定義する内容と定義しない内容を明確に区分する

### バリデーションの基本原則
- 早期検証・型安全性・ユーザーフレンドリー・自動正規化・一貫性の5原則を定義
- バリデーションのタイミングを表形式で整理する

| タイミング例 | 対象 | 実装場所 |
|----------|------|---------|
| ツール呼び出し前 | ツールパラメータ | Pydanticモデル |
| エージェント呼び出し前 | invocation_state | Pydanticモデル |
| データ読み込み後 | 外部データ | ツール内部 |

- ValidationError発生時の処理方針（エラー通知・再入力促進・ログ記録）を記載

### Pydanticモデルの使用方針
- 定義場所（例：`models/data_models.py`）と命名規則（例：`{用途}Input` / `{用途}Output`）を明記
- 基本構造のコード例を記載（`field_validator` 使用）

### データ型別バリデーション
本システムで使用するデータ型ごとにセクション（4, 5, ...）を作成すること：
- **日付**: 標準形式（YYYY-MM-DD）・申請期限チェック等
- **金額**: 数値型・上限チェック・承認判定等
- **文字列**: 必須チェック・長さ制限等
- **列挙型**: 業務要件で定義された区分値

各データ型セクションに含める内容：
- 基本原則（標準形式・自動変換・範囲チェック等）
- 標準形式・許容値の定義
- `field_validator` を使ったバリデーション実装例
- 業務ルールに基づく特殊ルール（申請期限・金額上限等）

### カスタムバリデーションルール
- 列挙型フィールドのバリデーションと正規化は以下の設計指針・実装パターンに従って記載すること
  - 列挙型フィールドは `Literal` 型または `Enum` で定義し、受け入れ可能な値を明示すること
  - LLMが渡す可能性のある表記バリエーション（別表記・略称など）を洗い出し、許容するかどうかを設計時に判断すること
  - 複数の表記を許容する場合は、内部処理用の正規化関数（`field_validator`）を定義し、受け入れ後に統一表記へ変換すること
  - 正規化後の値を後続処理で使用すること（入力値をそのまま使用しないこと）
- 業務ルールに基づくカスタムバリデーターを具体的なコード例で記載

### バリデーションエラーメッセージ
- わかりやすさ・具体性・一貫性の基本原則を定義
- 日本語での基本形式（`{フィールド名}のエラー: {エラー内容} / 対処方法: {対処方法}`）を定義
- 具体的なエラーメッセージ例を記載（ユーザーが対処できる内容）

### バリデーションのテスト
- 正常系・異常系・境界値・正規化・エラーメッセージの5観点でテストケースを表形式で記載
- テスト設計書への参照を明記
