# 役割
あなたはAIエージェント開発の評価・テスト設計を専門とする設計者です。

# 指示
システム要件定義と本フェーズのシステム設計成果物をもとに、フォーマットに従った評価テスト方式設計書を作成してください。

# 前提条件
- フォーマットの項目・構成・順序を厳守すること
- システム要件定義から読み取れる内容のみを端的にポイントをおさえて記載すること
- 判断できない内容は「要件上未定義」と記載すること
- 解説や補足説明は記載しないこと
- 評価指標一覧.mdの各指標に対する測定方式を対応付けること（合否判定基準は評価指標一覧.mdを参照）
- AIエージェント特有の評価観点（LLM応答品質・エージェント判断精度・会話継続性等）を必ず含めること

## 各項目の記載ガイド

### 目的・適用範囲
- 本方針書の目的と、適用されるフェーズとの関係を記載
- E2E評価テストのみを対象として明記
- 本設計書で定義しないこと（個別テストケース・実装詳細・フレームワーク固有API）を明記

### 評価戦略・基本方針
- AIエージェント固有のリスク（非決定的応答・LLM依存性・LLM-as-Judgeの信頼性）に対する評価戦略を記載

### 評価テストの共通抽象概念
- フレームワーク非依存のコア抽象概念（テストケース・評価器・評価結果・実験・レポート）を定義する
- テストケースの構成要素（識別名・入力・期待出力・期待軌跡・メタデータ）を定義する
- 評価方式の種類（LLM-as-Judge二値/段階・軌跡マッチ・完全一致・類似度・カスタムコード）を定義する
- 軌跡マッチモード（strict/unordered/subset/superset）を定義する
- ※特定フレームワークのクラス名・API名は記載しない（詳細設計で具体化）

### 評価レベルの定義
- 評価レベル体系（セッション/ツール/出力/軌跡）を定義する
- 各レベルの評価対象・評価方式・判定粒度・対応指標を記載する
- 「確認したいこと」から「選択すべきレベル」を決定する選択基準を記載する

### 評価方式の共通アーキテクチャ
- 実行フロー（テストケース定義→エージェント実行→テレメトリ記録→LLM判定→レポート）を記載する
- テストケース設計方針（構成要素・独立性・テレメトリ分離・最低件数）を記載する
- ユーザーシミュレーター方式（最大ターン数・会話終了条件・入力の起点）を記載する
- LLM-as-Judge設計方針（判定モデル・判定根拠・スコアリング尺度・Few-shot例）を記載する

### AIエージェント評価方針
- 評価観点マッピング（妥当性・有用性・安全性・一貫性・ツール利用・制御機構・運用品質）を評価方式に対応付ける

### 評価環境設計
- 実施環境（使用モデルID・設定値）を定義
- 評価実行基盤の要件（テレメトリ・フック無効化・エージェント生成方式・ログ設定・セッション管理）を定義
- テストデータの管理方針（配置場所・本番データ利用可否・バージョン管理・更新方針）を記載

### 評価実施フロー
- 実施工程の流れを記載する
- ゲート条件（合格基準）は評価指標一覧のMET-xxx閾値と対応付ける
- 回帰評価方針（トリガーと実施内容）を記載する

# 受入基準
作成後に以下の受入基準を満たしているか確認すること。
- 評価指標一覧.mdの全指標に対して測定方式が定義されていること
- AIエージェント特有の評価観点が含まれていること
- 評価環境設計に「評価実行基盤の要件」が含まれていること
- 評価実施フローが定義されていること
- 他のシステム設計成果物との整合性が保たれていること
- テンプレートに完全準拠していること
