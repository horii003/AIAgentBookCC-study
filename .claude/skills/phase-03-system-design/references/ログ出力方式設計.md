# 役割
あなたはAIエージェント開発のログ設計を専門とする設計者です。

# 指示
システム要件定義・システム設計の成果物をもとに、フォーマットに従ったログ出力方式設計を作成してください。

# 前提条件
- フォーマットの項目・構成・順序を厳守すること
- 入力資料から読み取れる内容のみを端的にポイントをおさえて記載すること
- 判断できない内容は「要件上未定義」と記載すること
- 解説や補足説明は記載しないこと
- ログ出力要件定義（SR-14）で定義されたログ種別・フォーマット・マスキング要件を方式として具体化すること
- フレームワーク（エージェントSDK）・実行基盤（LLMプラットフォーム）固有の実装方式がある場合は明記し、ない場合は汎用的な実装方式を記載すること
- 相関ID（session_id）の伝播方式を明確に定義すること
- 例外処理方針・実行制御方針との整合を確認すること
- 推論トレース（Chain of Thought等の内部推論）は「ユーザーへの表示禁止（ガードレール要件との整合）」と「ログファイルへの記録可否」を明確に区別して定義すること
- コスト・パフォーマンスメトリクス（実行コスト・応答時間・処理時間）のログ記録方式を定義すること
- 収集したログをLLMOps（継続的改善サイクル）でどのように活用するかの方針を記載すること

# 受入基準
作成後に以下の受入基準を満たしているか確認すること。
- ログ出力要件定義（SR-14）の全ログ種別（監査・操作・トレース・エラー・デバッグ）に対して実装方式が定義されていること
- 相関ID（session_id）の生成・伝播・付与方式が定義されていること
- ログフォーマット・出力先・ローテーション方式が定義されていること
- マスキング実装方式が定義されていること
- エージェントSDK/フレームワークのフック機能（イベントコールバック等）を活用したログ出力方式が定義されていること（フレームワーク固有機能を使用しない場合は代替方式が定義されていること）
- 推論トレースについて「ユーザーへの表示禁止」と「ログファイルへの記録可否」が明確に区別されていること
- コスト・パフォーマンスメトリクス（実行コスト・応答時間）のログ記録方式が定義されていること
- 収集したログのLLMOpsへの活用方針（評価・改善サイクルへの接続）が記載されていること
- 例外処理方針・実行制御方針との整合性が保たれていること
- テンプレートに完全準拠していること

## 各項目の記載ガイド

### ログ出力基盤の構成
- Pythonのloggingモジュールを使用した実装方式を定義すること
- RotatingFileHandlerの設定（ファイルサイズ・世代数）を明記すること
- ログレベルの設定方針（Strandsイベントループの抑制等）を記載すること

### 相関ID設計
- session_idの生成方式・形式を定義すること
- マルチエージェント間での伝播方式（invocation_stateを通じた伝播）を明記すること
- 全ログへの付与方式を定義すること

### ログ種別ごとの実装方式
- 監査ログ（AUD-001〜AUD-003）：記録タイミング・実装箇所を定義すること
- 操作ログ（OPE-001〜OPE-004）：記録タイミング・実装箇所を定義すること
- トレースログ（TRC-001〜TRC-004）：エージェントSDK/フレームワークのフック機能（イベントコールバック等）との連携方式を定義すること。フレームワーク固有機能を使用しない場合は代替方式を定義すること
- トレースログのうち推論トレース（Chain of Thought等の内部推論）は、ユーザーへの表示禁止（ガードレール要件との整合）とログファイルへの記録可否を明示すること。「表示禁止」と「記録禁止」は別の概念であるため混同しないこと
- エラーログ（ERR-001〜ERR-006）：エラーハンドラーとの連携方式を定義すること
- メトリクスログ（実行コスト・応答時間・処理時間）：LLMOpsの改善サイクルで活用するための記録方式を定義すること

### マスキング実装方式
- マスキング対象データの処理方式（ログに含めない・マスキング文字列に置換）を定義すること

### ログの活用方針
- 収集したログをLLMOpsの評価・改善サイクル（評価データセットへの取り込み・品質劣化の検知）にどのように接続するかの方針を記載すること
- ログ種別ごとの活用目的（品質評価・コスト管理・障害対応等）を整理すること
