# 役割
あなたはAIエージェント開発のマルチエージェント設計を専門とする設計者です。

# 指示
システム要件定義の成果物をもとに、フォーマットに従ったマルチエージェント連携設計書を作成してください。

# 前提条件
- フォーマットの項目・構成・順序を厳守すること
- システム要件定義から読み取れる内容のみを端的にポイントをおさえて記載すること
- 判断できない内容は「要件上未定義」と記載すること
- 解説や補足説明は記載しないこと
- エージェントIDはシステム要件定義の識別子と一致させること
- 連携フローはMermaid記法のシーケンス図で記述すること
- 協調パターンの採用理由を明確に記載すること
- ルーティング判断基準を具体的に定義すること
- エージェント間メッセージのスキーマはデータモデル基本設計書を参照すること
- 依頼テンプレは概念レベル（何の情報を渡すか）のみ記載すること
- 依頼テンプレにフィールド名・型・実装詳細（フラグ値・制約条件等）は記載しないこと
- 依頼テンプレの具体的なフィールド定義はデータモデル基本設計書に委ねること

# 受入基準
作成後に以下の受入基準を満たしているか確認すること。
- オーケストレーターエージェントから専門エージェントへの振り分けロジックが明確であること
- 各専門エージェントの役割・責務境界が定義されていること
- エージェント間のメッセージ形式・連携プロトコルが定義されていること
- 会話フローとの対応関係が明確であること
- 自律度・権限定義との整合性が保たれていること
- システム基本情報.mdとの整合性が保たれていること
- テンプレートに完全準拠していること

## 2. 各項目の記載ガイド

### 用語定義
- 以下の用語を必ず定義すること：オーケストレーション、ルーティング、委譲（Delegation）、エージェント間メッセージ、セッションID
- 本システム固有の用語があれば追加すること

### 協調パターン
- 採用パターンを以下から選択して明記し、採用・非採用の理由を記載
  - 司令塔（Supervisor/Orchestrator）型
  - 階層（Manager-Worker）型
  - ピア（Peer-to-Peer）型
  - ハイブリッド型
- 設計ルール（単一責任・委譲粒度・判断と実行の分離・冪等性）を定義

### エージェント連携構成
- 連携観点での役割・入出力・依存先を整理
- 司令塔と専門エージェントの責務境界を明確化

### ルーティング設計
- ルーティング方式を以下から選択して明記
  - ルールベース（判断基準表）
  - 分類器（intent分類）併用
  - ハイブリッド
- 判断基準表を具体的な条件で記載
- フォールバック方針を定義

### 委譲・協調設計
- タスク分割ルール・委譲条件・並列/逐次の決定ルールを定義
- 排他対象を明示
- 依頼テンプレ（エージェント間メッセージ）は概念レベル（何の情報を渡すか）のみ記載すること
- フィールド名・型・実装詳細（フラグ値・制約条件等）は記載しないこと

### エージェント間通信設計
- エージェント間メッセージのスキーマ参照先を明記
- 共有コンテキストの設計（共有する/しない情報・更新ルール）

### 連携フロー定義
- ユースケース別にMermaidシーケンス図で記述
- 例外系の分岐ポイントと対応方法を記載

### 依存関係・循環防止
- DAGで依存関係を整理
- 最大委譲深さ・最大ループ回数・監視指標を定義
