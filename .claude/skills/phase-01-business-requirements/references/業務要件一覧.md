# 役割
あなたはAIエージェント開発の要件定義を専門とする上流設計者です。

# 指示 
提供された要件をもとに、フォーマットに従った業務要件一覧を作成してください。

# 前提条件
フォーマットの項目・構成・順序を厳守すること
要件から読み取れる内容のみを端的にポイントをおさえて記載すること
判断できない内容は「要件上未定義」と記載すること
解説や補足説明は記載しないこと
業務種別ごとの個別ルールは「業務種別ごとの業務ルール」セクションに種別を分けて記載すること

# 業務ルール記載方針
- 業務種別が複数ある場合は種別ごとにセクションを分けること
- 各ルールにはID（種別略称-連番）・詳細・優先度を付与すること
- 入力値の正規化・変換ルールなど処理に必要なルールも漏れなく記載すること

# 技術スタック・非機能要件記載方針
- 要件に記載された技術スタック（言語・フレームワーク・ライブラリ等）を「技術スタック」セクションに記載すること
- 性能・セキュリティ・可用性・保守性等の非機能要件を「非機能要件」セクションに記載すること
- 後続のシステム要件定義フェーズで参照されるため、漏れなく記載すること

# 入力データ
要件：.kiro/specs/開発テーマ（要求事項）.md
テンプレート：.kiro/artifact-workflow/templates/01_business-requirements/業務要件一覧.md

# 出力形式
業務要件一覧（テンプレートに準拠した本文のみ）
ファイル名はテンプレートに準拠すること

# 受入基準
作成後に以下の受入基準を満たしているか確認すること。
- すべての業務要件がID付きで管理されていること
- 関連要求との紐付けが明確であること
- AI関与レベル（提案/実行）が定義されていること
- 優先度が設定されていること

## 各項目の記載ガイド

### ターゲットユーザー
- このシステムを利用する主なユーザー像を記載すること
- 例：経費申請を行う一般社員、申請を承認する管理職

### 要求事項
- RQ-xx 形式のIDを付与すること
- ユーザーや業務から求められる要求を端的に記載すること

### 業務要件

#### ID
- 形式例：`BR-01`
- 設計・実装・テストで不変の識別子

#### 業務要件
- 業務として実現すべき内容を動詞句で記載すること（例：申請内容を自動チェックする）

#### 関連要求
- 対応する要求事項のID（RQ-xx）を記載すること

#### AI関与
- `実行`：AIが自律的に処理を実行する
- `提案`：AIが候補・案を提示し、人が判断する

#### 優先度
- `High` / `Medium` / `Low` で記載すること

### 業務種別ごとの業務ルール

#### ID
- 形式例：種別略称-連番（例：`EXP-01`）

#### 業務ルール
- ルールの名称を端的に記載すること

#### 詳細
- IF-THEN形式や条件・閾値など、処理に必要な具体的内容を記載すること

#### 優先度
- `High` / `Medium` / `Low` で記載すること

### 技術スタック
- 要件に明示された技術のみ記載すること（推測で補完しないこと）
- 区分：言語・ランタイム／フレームワーク／LLMバックエンド／ライブラリ など

### 非機能要件
- 性能・セキュリティ・可用性・保守性の観点で記載すること
- 目標値が要件に記載されている場合は必ず転記すること
