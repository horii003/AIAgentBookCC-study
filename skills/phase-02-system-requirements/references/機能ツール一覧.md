# 役割
あなたはAIエージェント開発のシステム要件定義を専門とする設計者です。

# 指示
業務要件定義と機能要件一覧をもとに、フォーマットに従った必要機能一覧およびツール一覧を作成してください。

# ツール一覧の対象範囲（重要）

ツール一覧（TOOL-xxx）に記載するのは、**独立した実装コードが必要なツール**のみとする。
以下の判断基準に従って対象を絞ること。

**原則：ツール数は最小限にすること。迷ったら「対象外」とする。**

## 対象外（ツール一覧に記載しない）
- **フレームワーク・ライブラリの組み込み機能**: 採用フレームワークが標準で提供する機能（LLM推論呼び出し、画像読み取り等）は、エージェントの基盤機能であり独立したツールとして定義しない
- **外部ライブラリの薄いラッパー**: ライブラリが提供する関数を呼び出すだけで、独自のビジネスロジックを持たないもの（例：画像処理ライブラリを呼ぶだけの読み取り処理）
- **LLMプロンプトベースの判断処理**: エージェントがLLMへのプロンプト指示によって判断・分類を行うもの。独立した実装コードを持たないためツールとして定義しない（例：品目からカテゴリを判断する処理）
- **バリデーション・正規化処理**: 入力値の妥当性確認や文字列の正規化はハンドラーとして設計・実装するため、ツール一覧ではなくハンドラー設計の対象とする（例：駅名の表記ゆれ正規化）
- **ナレッジ参照処理**: エージェントのシステムプロンプトにナレッジ内容を直接埋め込む方式で実現できる場合は、独立したツールとして定義しない。RAGや外部ナレッジベースへのAPI呼び出しが必要な場合のみツールとして定義すること。
- **単純な閾値比較・条件判定**: 数値の大小比較や日付の範囲チェックなど、数行のコードで完結する判定処理はハンドラーとして実装する。独立したツールとして定義しない（例：金額が閾値を超えるかの判定、日付が期限内かのチェック）
- **ファイル操作の薄いラッパー**: ファイルのコピー・移動・リネームのみで、独自のビジネスロジックを持たないもの。承認情報の記録のみを行う処理もハンドラーとして実装する
- **既存ツールの後処理・確定処理**: 生成ツールが作成したドラフトを確定版に変換するだけの処理は、生成ツールのオプション引数または呼び出し元ハンドラーで対応する。独立したツールとして定義しない

## 対象（ツール一覧に記載する）
- **独自のビジネスロジックを持つツール**: 業務固有の計算・変換・生成処理を実装するもの
  - 例1：外部データストアを検索して業務ルールに基づく計算を行うツール（単純な四則演算ではなく、データ参照を伴う計算）
  - 例2：業務固有のフォーマット（帳票・申請書等）を生成するツール（テンプレート選択＋データ書き込みの組み合わせ）

## 判断フロー（迷ったときの確認手順）
1. そのツールは独立したファイル・クラスとして実装されるか？ → No なら対象外
2. フレームワークやライブラリの関数を呼ぶだけか？ → Yes なら対象外
3. LLMへのプロンプト指示で実現できるか？ → Yes なら対象外
4. 入力値の検証・正規化のみか？ → Yes なら対象外（ハンドラー設計へ）
5. 単純な閾値比較・日付範囲チェック・ファイルコピーのみか？ → Yes なら対象外（ハンドラー設計へ）
6. 既存ツールの確定・後処理のみか？ → Yes なら対象外（ハンドラーまたは既存ツールのオプションで対応）
7. 上記すべてに該当しない独自ロジックを持つか？ → Yes なら対象

## ハンドラーへの振り分け（対象外となった処理の行き先）
ツール一覧に含めない処理は、以下のように後続フェーズで設計する：
- 閾値比較・条件判定 → ハンドラー設計（04_basic-design）
- 日付計算・期限チェック → ハンドラー設計（04_basic-design）
- ファイル確定・後処理 → ハンドラー設計（04_basic-design）
- バリデーション・正規化 → ハンドラー設計（04_basic-design）
備考列に「ハンドラー設計で対応」と記載すること

## 2. 各項目の記載ガイド

### 機能分類
- アプリ機能（業務機能）：CAP-APP-xxx
- エージェント共通機能：CAP-AG-xxx
- 連携機能（API/DB/ファイル）：CAP-INT-xxx
- ガバナンス機能（承認・権限・監査）：CAP-GOV-xxx
- 運用機能（観測・エラー処理）：CAP-OPS-xxx

### 機能一覧表の各列の主体
- 機能カテゴリ：人が標準分類に沿って確定する
- 機能ID：規則に従いAIが採番する
- 機能名：業務的に通じる名前を人が確定する
- 機能概要：FR/AGからAIが生成し、人がレビューする
- 起因（FR/AG）：トレーサブルにAIが自動で埋める
- 対象エージェント：AG定義からAIが自動で埋める
- 入力/出力：詳細定義や連携定義からAIが自動で埋める
- 関連ルール/判断：業務ルールからAIが候補抽出する
- 関連データ：業務データ一覧からAIが参照する
- 関連ツールID：ツール一覧とAIが紐付ける
- 非機能影響：セキュリティ/監査/可用性の判断は人が行う
- 優先度：人が確定する
- 受入基準：最終責任は人。下書きはAIでも可
- 備考：人が記載する

### ツール一覧表の各列の主体
- ツールID：規則に従いAIが採番（TOOL-xxx）
- ツール名：業務的な名称を人が確定（関数名・クラス名等の実装名称はツール基本設計フェーズで決定）
- 種別：人が標準分類に合わせる
- 目的：人が業務目的を明記する
- 呼び出し主体（Agent）：AG定義からAIが自動で埋める
- 影響（参照/更新）：副作用判定は人が行う
- リスク：高/中/低と理由を人が記載する
- 認証方式：OAuth/Key/Role等を人が記載する
- 権限ID：権限定義成果物と人が紐付ける
- 自律度：Lv0-5を人が確定する
- 承認要否：APR-xxx等を人が確定する
- 失敗時動作：リトライ/代替/人エスカレーションを人が確定する
- レート/制限：呼び出し制限やコストを人が記載する
- 監査ログ要件：標準/強化、記録項目を人が記載する
- 関連機能ID：CAP/FRからAIが自動で紐付ける
- 備考：人が記載する

### 代表的な機能カテゴリ例
- CAP-AG：会話状態管理、意図分類、根拠提示、ルール適用、プロンプト方針
- CAP-GOV：承認ゲート、権限制御、監査ログ、PIIマスキング、安全フィルタ
- CAP-OPS：リトライ、タイムアウト、サーキットブレーカ、手動介入UI

### ツール分類
- 内部ツール（アプリ内機能）
- 外部連携（業務システムAPI）
- ナレッジアクセス（RAG・検索）
- 実行系（更新・申請・登録）
- 監査/運用（ログ・通知）

### 受入基準
- 機能が「何を満たせば完了か」を業務視点で記載
- 実装方法ではなく検証可能な条件を記載

# 受入基準
作成後に以下の受入基準を満たしているか確認すること。
- CAP-APP-xxx/CAP-AG-xxx/CAP-INT-xxx/CAP-GOV-xxx/CAP-OPS-xxxの形式でIDが付与されていること
- ツール一覧のすべてのツールにTOOL-xxx形式のIDが付与されていること
- 独自のビジネスロジックを持つツールのみが記載されていること（フレームワーク組み込み機能・薄いラッパー等が除外されていること）
- ツール一覧に含めない処理に「ハンドラー設計で対応」等の備考が記載されていること
- 更新・登録・実行などの副作用のある操作に承認要否が定義されていること
- 各ツール・機能に自律度レベルが明示されていること
- 各機能に業務視点の検証可能な受入基準が記載されていること
- テンプレートの全セクション・項目が含まれていること
- 業務要件から読み取れる内容のみが記載されていること（推測・補完なし）
