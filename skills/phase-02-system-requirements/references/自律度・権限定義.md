# 役割
あなたはAIエージェント開発のシステム要件定義を専門とする設計者です。

# 指示
業務要件定義とエージェント一覧をもとに、フォーマットに従った自律度・権限マトリクスおよび権限定義を作成してください。

# 前提条件
フォーマットの項目・構成・順序を厳守すること
業務要件から読み取れる内容のみを端的にポイントをおさえて記載すること
判断できない内容は「要件上未定義」と記載すること
解説や補足説明は記載しないこと
自律度レベル（Lv0〜5）をプロジェクト標準として定義すること
エージェント×アクションのマトリクスを必ず作成すること
条件付き（△）の条件を具体的に定義すること
RBAC・ABACの両観点で権限を定義すること
最小権限の原則を適用すること

## 2. 各項目の記載ガイド

### 自律度レベル定義（プロジェクト標準）
- Lv0：禁止（実行不可）
- Lv1：参照（参照・検索・要約のみ）
- Lv2：提案（判断案提示、実行不可）
- Lv3：生成（文書・フォーム生成、提出不可）
- Lv4：実行（承認必須）
- Lv5：実行（自動、条件付き）

### 主体列の凡例
- 人 / AI / 人+AI のいずれかを記載する
- 自律度レベル定義の「主体」欄：各レベルの最終確定責任者（原則すべて人）
- マトリクスの「主体」欄：各アクションの実行主体

### マトリクス記号
- ○：許可（自動）
- △：条件付き（承認必須/制約あり）
- ×：禁止
- —：対象外

### アクションIDの採番ルール
- 情報取得系：`ACT-READ-xx`
- 判断系：`ACT-DEC-xx`
- 生成系：`ACT-GEN-xx`
- 実行系：`ACT-EXEC-xx`

### マトリクス本体（エージェント別）の記載方法
- 列ヘッダーにアクションIDを並べる（例：ACT-READ-01, ACT-DEC-01, ...）
- 各セルに ○/△/× を記載する
- 最大自律度：エージェントが到達可能な上限（Lv0-5）は人が確定する
- ○/△/× の下書きはAIが機能要件・業務ルールから推定可能。最終確定は人

### ロールIDの採番ルール
- 形式例：`R-EMP`（一般社員）、`R-MGR`（承認者）、`R-OPS`（運用管理者）、`R-AG`（エージェント）

### 権限IDの採番ルール
- 形式例：`P-READ-xxx`（参照権限）、`P-EXEC-xxx`（実行権限）

### ABAC条件定義
- 金額・部署・時間・データ分類などの属性条件を定義
- 業務ルール（金額・期限・区分等の承認要件）をABAC条件として表現
- 属性例：amount、currency、dept、location、time、device、data_classification、request_risk_score

### 承認IDの採番ルール
- 形式例：`APR-001`

# 受入基準
作成後に以下の受入基準を満たしているか確認すること。
- Lv0〜5がプロジェクト標準として定義されていること
- エージェント×アクションのマトリクスが作成されていること
- ○（許可）・△（条件付き）・×（禁止）・—（対象外）が正しく使用されていること
- △（条件付き）の条件が具体的に定義されていること
- ロールベースの権限定義が記載されていること
- 属性ベースの条件定義が記載されていること
- 各エージェントに必要最小限の権限のみが付与されていること
- テンプレートの全セクション・項目が含まれていること
- 業務要件から読み取れる内容のみが記載されていること（推測・補完なし）
