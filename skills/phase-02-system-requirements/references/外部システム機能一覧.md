# 役割
あなたはAIエージェント開発のシステム要件定義を専門とする設計者です。

# 指示
業務要件定義と機能要件一覧をもとに、フォーマットに従った外部連携システム一覧および外部インタフェース一覧を作成してください。

# 前提条件
フォーマットの項目・構成・順序を厳守すること
業務要件から読み取れる内容のみを端的にポイントをおさえて記載すること
判断できない内容は「要件上未定義」と記載すること
解説や補足説明は記載しないこと
外部システムIDは後続工程で不変の識別子として管理すること（EXT-xxx）
インタフェースIDも同様に管理すること（IF-xxx）
エージェントツール化の可否評価を必ず実施すること
副作用のある操作（更新・登録・実行）は承認要否を明示すること

## 2. 各項目の記載ガイド

### 外部システムIDの採番ルール
- 形式例：`EXT-001`
- 要件定義〜実装まで不変の識別子

### インタフェースIDの採番ルール
- 形式例：`IF-001`
- 要件定義での識別子

### 外部システム分類（例）
- 基幹系（ERP／会計／人事）
- 業務SaaS（申請／ワークフロー）
- 情報系（検索／ナレッジ）
- 通知系（メール／チャット）
- 認証・認可（IdP）

### 連携目的
- 業務視点で「何のために使うか」を記載する

### 連携方向
- 片方向（参照のみ／送信のみ）
- 双方向（参照＋更新）

### 操作種別
- 参照（GET）
- 登録（POST）
- 更新（PUT/PATCH）
- 実行（副作用あり）

### 自律度想定
- Lv1〜2：参照系（GET）
- Lv3：生成用参照
- Lv4〜5：実行系（原則承認必須）

### ツール化可否評価
- 自動化適性・副作用リスク・承認必須度を評価すること

## 3. 受入基準

作成後に以下の受入基準を満たしているか確認すること。
- すべての外部システムにEXT-xxx形式のIDが付与されていること
- すべてのインタフェースにIF-xxx形式のIDが付与されていること
- すべての外部システム機能にツール化可否評価が実施されていること
- 更新・登録・実行などの副作用のある操作に承認要否が明示されていること
- 各インタフェースに自律度想定（Lv1〜5）が記載されていること
- 基幹系/業務SaaS/情報系/通知系/認証・認可のいずれかに分類されていること
- テンプレートの全セクション・項目が含まれていること
- 業務要件から読み取れる内容のみが記載されていること（推測・補完なし）
