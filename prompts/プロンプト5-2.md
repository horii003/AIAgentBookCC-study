## 2. Human-in-the-Loop 承認フック

### 承認コールバック仕様

#### シグネチャ

- 引数: `tool_name`（str）、`tool_params`（dict）
- 戻り値: `tuple[bool, str]`

#### 戻り値

- `(True, "")` → OK（何もしない → ツール実行される）
- `(False, "修正内容")` → 修正要望（`event.cancel_tool` にメッセージをセット → ツールキャンセル）
- `(False, "CANCEL")` → キャンセル（`event.cancel_tool` にキャンセルメッセージをセット → ツールキャンセル）

#### ツール中止方法

- `event.stop_reason` ではなく `event.cancel_tool` にメッセージ文字列をセットする

> 設計書内でこの仕様に関連する記述（シグネチャ・戻り値・処理フロー・コード例・テスト観点等）をすべて洗い出し、上記仕様と一致するよう修正すること。

## 6. LoopControlHook 仕様

### ループ上限到達時の処理

- 上限到達時は `raise LoopLimitError` でエラーを発生させる
- カウントは `AfterModelCallEvent` で行うが、`event.exception` が存在する場合はスキップする

### LoopLimitError の定義場所

- `LoopLimitError` はカスタム例外クラスとして定義すること
- 定義するモジュールを設計書内で明示し、依存関係・使用例・テスト観点など全セクションで統一して記載すること

### ログ出力イベント（ループ制御用イベントに加え、以下のログ出力用イベントも登録する。）

- BeforeModelCallEvent: ループ回数を INFO レベルでログ出力
- BeforeToolCallEvent: ツール名を INFO レベルでログ出力
- AfterToolCallEvent: ツール名を INFO レベルでログ出力

### その他仕様

- AfterInvocationEvent のハンドラーでは「合計ループ回数のログ出力（INFO）」を行う。（ループカウンタのリセットは行わない）
- BeforeInvocationEvent のハンドラーではループカウンタのリセットは行う
- LoopLimitError は current_iteration・max_iterations・agent_name の3フィールドを保持する
- LoopLimitErrorを raise する場合、定義済みの全フィールドを引数として渡すこと
- ループ上限到達時のログレベルは WARNING とする

## 7. ErrorHandler クラス仕様

- クラスのメソッドの責務は「ユーザー向けメッセージ文字列の生成と返却」のみとする
- クラスの責務にログ出力・セッション状態更新を含めない
- コンストラクタの引数は持たない（SessionManager 等の依存注入は行わない）
- コンストラクタに `logger` インスタンス変数を定義しない
- 外部ライブラリ依存として `logging` を列挙しない
- 全メソッドの引数は `例外オブジェクト` のみとする
- 各メソッドの戻り値は「ユーザー向けメッセージ文字列（str）」のみとする
- 各メソッド内でのセッション状態更新（SessionManager の呼び出し）は禁止する
- 各メソッド内でのログ出力（logging.error / logging.warning 等）は禁止する
- ログ出力は呼び出し元モジュール（各エージェント・各ツール）が ErrorHandler 呼び出し前に実施する

### メソッド仕様

- ErrorHandlerメソッド定義を、以下の仕様にする。
  - handle_throttling_error: 対象例外 ModelThrottledException、用途 APIレート制限
  - handle_max_tokens_error: 対象例外 MaxTokensReachedException、用途 最大トークン数到達
  - handle_context_window_error: 対象例外 ContextWindowOverflowException、用途 コンテキストウィンドウ超過
  - handle_fare_data_error: 対象例外 FileNotFoundError / Exception、用途 運賃データ読み込み失敗
  - handle_calculation_error: 対象例外 Exception、用途 運賃計算失敗
  - handle_file_save_error: 対象例外 IOError / PermissionError / Exception、用途 Excelファイル保存失敗
  - handle_validation_error: 対象例外 ValidationError、用途 Pydanticバリデーション失敗
  - handle_keyboard_interrupt: 対象例外 KeyboardInterrupt、用途 ユーザーによる中断
  - handle_loop_limit_error: 対象例外 LoopLimitError、用途 ループ上限到達
  - handle_runtime_error: 対象例外 RuntimeError、用途 その他の実行時エラー
  - handle_unexpected_error: 対象例外 Exception、用途 予期しないエラー

## 11. 交通費計算ツール エラー時戻り値

- データ読み込み系メソッドのエラー時戻り値は `tuple[bool, str]` 型の `(False, エラーメッセージ文字列)` とする
- 計算系メソッドのエラー時戻り値は `{"success": False, "message": エラーメッセージ文字列}` の dict 形式とする
- 戻り値形式は、以下の全セクションで統一して記載すること

  - エラー処理フロー（フロー図内の戻り値）

- 例外を raise せず、必ず戻り値としてエラー情報を返す（補足情報・実装上の注意点セクションも統一した記載をすること）
- FileNotFoundError のログレベルは WARNING とする

  - 例外テーブル（発生条件・戻り値列）

## 14. 申請受付窓口エージェントの例外捕捉

以下の例外を個別に捕捉し、ErrorHandler でユーザー向けメッセージを生成して print で表示する。
- `KeyboardInterrupt`：INFO レベルでログ出力。ループを終了する（break）
- `LoopLimitError`：WARNING レベルでログ出力。ループを継続する（continue）
- `ContextWindowOverflowException`：WARNING レベルでログ出力。ループを継続する（continue）
- `MaxTokensReachedException`：WARNING レベルでログ出力。ループを継続する（continue）
- `RuntimeError`：ERROR レベル（exc_info=True）でログ出力。ループを継続する（continue）
- `Exception`（その他）：ERROR レベル（exc_info=True）でログ出力。ループを継続する（continue）
- 戻り値はなし（対話ループのため不要）
- ログにはセッションIDを付加情報として含める
- 上記の継続/終了の区別は、エラーハンドリングテーブル・ログ出力テーブル・テスト観点の全箇所で統一して記載すること

## 15. 専門エージェントの例外捕捉

以下の例外を個別に捕捉し、それぞれエラーメッセージ文字列を返す。
- `LoopLimitError`：WARNING レベルでログ出力
- `ContextWindowOverflowException`：WARNING レベルでログ出力
- `MaxTokensReachedException`：WARNING レベルでログ出力
- `RuntimeError`：ERROR レベル（exc_info=True）でログ出力
- `Exception`（その他）：ERROR レベル（exc_info=True）でログ出力
- エラー時の戻り値はすべて `str`（呼び出し元 LLM コンテキストに渡されるため）
- ログにはクエリの先頭 50 文字を付加情報として含める

## 16. ファイル読み込み実装方針

- ファイル存在チェックは `os.path.exists()` による事前チェックで実装すること
- `open()` 実行時の `FileNotFoundError` 捕捉との併用は禁止し、事前チェックを優先する

## 17. セッションID 生成方法

- session_id の形式・採番ルールを以下のとおり定義する
  - 形式: {タイムスタンプ（秒単位）}_{UUID（8文字）}
  - 例: 20260323153045_a1b2c3d4
  - 生成方法: datetime.now().strftime("%Y%m%d%H%M%S") + "_" + uuid.uuid4().hex[:8]
  - 生成タイミング: セッション開始時にAG-001が生成する
